# Codex 画像生成の認証確認・復旧

このスキルは、`server/codexImageClient.mjs`（`generateImageWithCodexAppServer` / `generateTextWithCodexAppServer`）経由でLP・バナー・サムネ等の画像を生成する際、Codexのログイン状態を確認し、トークン失効時に最小限のやり取りで復旧するために使う。

## 前提: 認証ファイルの場所は環境によって変わる

`codex` CLIの認証情報（`auth.json`）は、シェルに `CODEX_HOME` 環境変数が設定されているかどうかで保存場所が変わる。

```bash
echo "$CODEX_HOME"
```

- `CODEX_HOME` が設定されている場合 → 実際に使われる認証ファイルは `"$CODEX_HOME/auth.json"`
- 未設定の場合 → デフォルトの `~/.codex/auth.json`

**サンドボックス環境（Bashツール経由）とユーザーが直接操作するターミナルとで、この環境変数の有無が異なることがある。** ログイン状態を確認・判定するときは、必ず自分（エージェント）のBash実行環境で `echo $CODEX_HOME` を確認し、そちらのパスの `auth.json` の更新時刻を見ること。`~/.codex/auth.json` だけを見て「ログインできていない」と誤判定しないよう注意する。

`codex login status` の出力（`Logged in using ChatGPT`）は、必ずしも「今まさに有効なトークンを持っている」ことを保証しない。判定は **`auth.json` の更新時刻（mtime）が直近の再ログイン操作後に進んでいるか** で行う。

## 症状: リフレッシュトークン失効エラー

画像生成中に以下のエラーが出ることがある。

```
Error: Your access token could not be refreshed because your refresh token was revoked. Please log out and sign in again.
```

**原因の傾向:**
- アクセストークンは数時間で期限切れになる。
- 複数の並列エージェント/フォークが同じ認証情報を共有して同時に `codex app-server` を起動していると、トークン更新（refresh）が競合し、一部のプロセスだけ「失効」扱いになることがある（リフレッシュトークンがローテーション式のため）。
- 長時間（数時間規模）の一括画像生成バッチの途中で、自然に期限切れになることもある。

## 復旧手順（ユーザーへの依頼は最小限に）

1. **`codex login` はブラウザでの対話的なOAuth認証が必要なため、エージェント（Bashツール）から直接実行できない。** 必ずユーザー自身に、チャット欄で `!` プレフィックスを付けて実行してもらう。

   ```
   !codex logout && codex login
   ```

2. 依頼は1回にまとめる。複数のLP/セクションが同時に認証エラーで止まっていても、まとめて「今、認証エラーで止まっている作業が○件あります。お手数ですが1回だけ以下を実行してください」と伝え、個別に何度も催促しない。

3. ユーザーから「やりました」「ログインできています」等の曖昧な返答があった場合は、まず自分で `auth.json` の更新時刻を確認する。ただし `CODEX_HOME` の場所を見ていることを再確認すること（上記の前提を参照）。**更新時刻が進んでいなければ、素直に「まだ反映されていないようです」と伝え、実際のターミナル出力（`Successfully logged in`等）を貼ってもらうよう頼む。** 曖昧な自己申告だけで復旧したと判断しない。

4. 復旧を確認できたら、止まっていた全タスクを一度に再開する（フォーク/サブエージェントへの再開メッセージをまとめて送る）。

## 並列実行時の注意

- 多数のサブエージェントを並列で走らせて画像生成をさせる場合、全員が同じ認証情報を共有するため、トークン切れは同時多発的に起こり得る。復旧は1回のログインで全員分をカバーできるので、慌てて何度も再ログインを頼まない。
- サブエージェント/フォークのトランスクリプトが失われて `SendMessage` で再開できないことがある（"No transcript found"）。その場合は無理に再開を試みず、新しいエージェントに「どこまで完了しているか」を明示した自己完結プロンプトで続きを依頼する。
