# 並列エージェント運用

このスキルは、LPデザインリポジトリで実装、調査、検証、レビューを行うときに使う。
目的は、サブエージェントと並列ツール実行で待ち時間を減らし、作業を早く終わらせること。

## 基本方針

- 作業開始時に、直列で進めるべき作業と並列化できる作業を分ける。
- 読み取り調査、差分レビュー、ブラウザ確認、テスト確認はサブエージェントへ渡しやすい。
- 編集を任せる場合は、担当ファイルを明確に分ける。
- 同じファイルを複数人で同時編集しない。
- 本体エージェントは、次の一手を止める作業を自分で進める。
- サブエージェントの結果は鵜呑みにせず、最終統合時に必要な部分だけ確認する。

## 使い分け

### explorer に渡す作業

- 既存構成の調査
- 変更差分のレビュー
- バグ候補や不足テストの洗い出し
- UI/UX、アクセシビリティ、動作確認の観点整理

### worker に渡す作業

- 担当ファイルが分かれている小さな実装
- ドキュメント追加
- テスト追加
- 既存実装と衝突しない補助スクリプト追加

worker に渡すときは、必ず次を伝える。

- 担当ファイル
- 他の作業者がいる前提で、他人の変更を戻さないこと
- 変更したファイルを最後に報告すること

## 並列化してよい例

- 本体がGUI実装を進め、explorerが既存LPファイル構造を調べる。
- 本体がAPIを直し、explorerがブラウザエラーと差分を確認する。
- worker A が `src/`、worker B が `server/`、本体が `skills/` を担当する。

## 並列化しない例

- 直後の編集判断に必要な調査。
- 同じファイルの同じ箇所を触る作業。
- Git commit、push、Xserver公開、Vercel公開など、順番と確認が重要な作業。
- ユーザー承認が必要な操作。

## 最初の手順

1. 目的を1文で整理する。
2. 作業を「本体で進めること」「サブエージェントに渡すこと」に分ける。
3. サブエージェントへ具体的で短い依頼を出す。
4. 待っている間に、本体は別の作業を進める。
5. 結果を統合して、ビルド、テスト、ブラウザ確認を行う。
