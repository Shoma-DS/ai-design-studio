# LP レスポンシブ対応

このスキルは、`lp-creator` で作ったLP（セクション画像を並べたラスターLP）を、PC幅とスマホ幅の両方できちんと読めるようにするために使う。

**新規LPを作るときは、`lp-creator` の手順の最後に必ずこのスキルを適用する。** 「レスポンシブにして」と明示的に頼まれたときだけでなく、LP制作そのものの標準工程に含める。

---

## 絶対原則（2026-07-17ユーザー確定）

スマホ版画像の流用・単純拡大縮小は禁止。**セクションごとに必ず次の2種類を別々に作成する。**

- **スマホ版セクション画像** — スマホ閲覧を前提とした縦長レイアウト
- **PC版セクション画像** — PC閲覧を前提とした横長レイアウト

内容・文章・ブランドイメージ・配色は共通でよいが、以下はPC版とスマホ版でそれぞれ最適化すること:

- 要素配置
- 文字サイズ
- 余白
- 画像サイズ
- CTA配置
- 情報の見せ方

PC版は**スマホ版を横に広げた見た目にしない**。PC画面の横幅を活かした構図として再設計する（詳しい判定基準・禁止パターンはStep 1参照）。

**完成条件:** PC版タブとスマホ版タブを並べたとき、同じ内容でも構図や情報配置の違いが一目で分かる状態になっていること。この条件を満たさないセクションが1つでもあれば未完成として扱う。

## 前提: PC版とスマホ版は「セクションごとに別々に作る、別レイアウトのLP」

**2026-07-16からユーザー承認済みの標準構造。** 以前は全セクション画像を縦結合して `lp-full.png` / `lp-full-mobile.png` の2枚にまとめ、ページ全体を1つの `<picture>` で丸ごと切り替えていたが、この「結合」方式は廃止した。

**新方式: `lp/index.html` はセクションの数だけ `<picture>` を並べる。** 1セクション＝1つの `<picture>` で、その中に「PC用画像」と「スマホ用画像」という**それぞれ個別に作成された別レイアウトの画像**を持つ。結合画像（`lp-full.png` 等）は作らない。

```html
<picture class="lp-section lp-section-hero">
  <source media="(max-width: 767px)" srcset="./images/mobile/01-hero.png">
  <img class="lp-section-img img-ratio-hero" src="./images/01-hero.png" alt="...">
</picture>
<picture class="lp-section reveal">
  <source media="(max-width: 767px)" srcset="./images/mobile/02-xxx.png">
  <img class="lp-section-img img-ratio-xxx" src="./images/02-xxx.png" alt="" loading="lazy">
</picture>
```

参考実装: `portfolio/astella-vxr-suv-lp/lp/`（index.html・style.css・scripts/generate-pc-mobile-first.mjs）。新規に作る・作り直すときはこのプロジェクトを構造の見本にする。

テキストは画像に焼き込まれているため、ブラウザの標準的なテキスト折り返しやフォントサイズ調整は効かない。**レスポンシブ対応＝画面幅に応じて、セクションごとに適切なレイアウトの画像に切り替えること。**

ポートフォリオ側（`gallery/` のプレビューモーダル）には既に「PC版／スマホ版」切り替えタブがある（`gallery/script.js` の `setDevice`）。これはiframe幅を変えて実URLの `<picture>` のメディアクエリを発火させる仕組みなので、**gallery側の改修は不要。** 各LP側がセクションごとに正しくPC専用・スマホ専用の画像を持ってさえいれば、タブ切り替えは自動的に正しく機能する。

**スマホ版が正本、PC版はスマホ版から派生させる（2026-07-11からユーザー承認済み、セクション分割後も変わらず）。**

- **縦横比の基本（2026-07-17ユーザー確定）: PC版は16:9前後の横長構図、スマホ版は9:16前後の縦長構図を基本とする。** 内容量が極端に多い・少ないセクションでは多少の前後は許容するが、「セクションの内容量に応じて縦横比を自由に決める」のではなく、まずこの比率を基準に構図を組み立てる。
- **PC版の構図は、スマホ版の情報設計（要素の並び順・余白のリズム・視線導線・アイコンや挿絵のスタイル）を基準にする。** スマホ版を単純拡大したのでも、まったく別のレイアウト言語に作り替えたのでもなく、「スマホ版の世界観を保ったまま、16:9前後の横幅を活かして要素配置を組み替えたPC版」を目指す。
- **PC版コンテナは `max-width: 1200px` で中央配置**（旧: `width: min(100%, 1080px)`）。1200px以上の画面では左右に余白が出る。横スクロールは発生させない。
- `server/codexImageClient.mjs` のデフォルトプロンプトは、この方針（スマホ版基準・1200px上限・PC=16:9前後／スマホ=9:16前後）に合わせて更新すること。

## Step 1 — 再生成が必要かどうかを判定する

全セクションを機械的に作り直す前に、まず数枚のセクション画像を実際に見て判断する（`Read`ツールで画像を開く）。

判定基準（PC版）— **この判定を誤ると「レスポンシブ切り替えは動くが中身は作り直されていない」状態になる。厳密に適用すること。**

PC版とスマホ版を並べて、**ナビ・見出し・本文・写真・カードなど主要要素の相対配置（左右関係・横並びか縦積みか）が一致しているかどうか**を見る。

- **配置が一致している（見出しが常に写真の上、カードが常にPC/スマホとも縦1列、等）** → たとえ縦横比や余白量が違っていても、それは「作り直し」ではなく「サイズ変更のみ」。**再生成が必要。**
- **配置が入れ替わっている（スマホでは縦積みの要素がPCでは左右横並びになっている、スマホで2列グリッドの要素がPCで横一列になっている等）** → 既に再設計済み。再生成は不要。

判断の目安: 「このPC画像から見出しか写真のどちらかを取り除いても、スマホ版のスクリーンショットとほぼ同じシルエットに見えるか？」→ Yesなら未対応（サイズ変更のみ）。

**特に見落としやすいのがヒーロー（1セクション目）。** 「ヒーローはブランドの世界観を保つため大きく変えない」という判断は誤り。ヒーローも他セクションと同じ基準で、PCの横幅を使って構造を変える（例: スマホでは見出しが写真の上にあるならPCでは見出しを写真の横に置く）。

判定基準（スマホ版）:

- **既に1080px前後の縦長・単一カラムで、見出し・本文とも画面幅に対して十分大きい** → 既にスマホ向けレイアウトなので、**再生成は不要**。無駄なAI生成コストと時間をかけない。
- **左右2カラム（テキスト+画像を横並び）、横一列に3つ以上のカード/アイコン、比較表など、PC画面前提の横方向レイアウトが実際にある** → 該当セクションのみスマホ版を再生成する。

判断はセクションごとに行ってよい。

## PC版をスマホ版基準で作り直す

1. **スマホ版セクション画像を参照画像として使う**（`lp/images/mobile/<section.imageName>` があればそれ、無ければ既存のスマホ寄りの縦長画像でよい）。PC版の古い横長画像は参照にしない。
2. `buildPrompt(section)` の結果に、以下のPC版レイアウト指示を追記してプロンプトを作る。

```
────────────────
【PC版レイアウト再構成の指示（最優先・厳守）】
このセクションは【PC/デスクトップ表示専用のセクション画像、max-width 1200pxのコンテナに中央配置される】として出力する。
・添付する参照画像（同一セクションのスマホ版）のコピー・ブランド要素・配色・情報量・視線導線の「世界観」は踏襲するが、**要素の配置（レイアウト構造）はスマホ版のまま流用しない。PCの横幅を使って組み替えること。**
・具体的には次のいずれかに該当する要素があれば、必ず配置を変える:
  - 見出し・テキストブロックが写真の上または下にある → PCでは写真の横（左右どちらか）に並べる
  - カード・アイコン・特徴ブロックが3つ以上、縦1列または2列グリッドで並んでいる → PCでは横一列、または3列以上のグリッドに並べる
  - 「テキスト→写真→テキスト」のように縦に交互配置されている → PCでは左右2カラムに分割する
・単に用紙のサイズ・余白比率だけを変えて「スマホ版と同じ配置のまま少し横に広い版」を作ることは禁止。それは「作り直し」ではなく「サイズ変更」であり、この指示の目的に反する。
・縦横比は16:9前後の横長を基本にする（横1536×縦1024px程度）。内容量が極端に多いセクションのみ、多少縦に伸びることを許容する。
・コピー・ブランド要素・配色・情報量はスマホ版と完全に同じに保つ（情報を削らない・足さない）
・1200px幅で表示したときに間延びせず、スマホ版と一貫した世界観に見える構図にする
・上下端は前後セクションへ自然につながる余白で終える
```

3. 生成した画像は必ず`Read`で開き、(a) スマホ版と要素の配置が入れ替わっているか（横並び化・グリッド化されているか。単なる縦横比違いになっていないか） (b) スマホ版と印象が近いか (c) 1200px幅で不自然に間延びしていないか (d) コピーが欠落・見切れていないか (e) 次セクションの要素が写り込んでいないか を確認する。(a)を満たさない場合は再生成する。
4. 古いPC版画像は上書きせず `references/pre-<変更内容>/` などに退避してから差し替える（後で見比べられるようにする）。
5. `lp/style.css` の `.lp-shell` を `max-width: 1200px; margin: 0 auto; width: 100%;` にする（`width: min(100%, 1080px)` などの古い指定から更新する）。セクション画像自体は `width: 100%; max-width: 1200px; height: auto;` で縦横比を保ったまま表示する。

## Step 2 — スマホ版セクション画像を生成する（無ければ先にこちらを作る）

**スマホ版は正本なので、`buildPrompt(section)` から直接生成する**（PC版からの変換ではない）。既に `lp/images/mobile/<section.imageName>` があり、内容量に対して十分読みやすい縦1カラムなら再生成不要（Step 1参照）。

再生成が必要なセクションについて、`server/codexImageClient.mjs` の `generateImageWithCodexAppServer` を使う（`skills/common/codex-image-auth/SKILL.md` の認証確認手順に従う）。

`buildPrompt(section)` の結果に、以下のスマホ向け指示を追記してプロンプトを作る。既存のPC版画像がある場合は参照画像として渡してよい（配色・ブランド・コピーの一貫性を保つため）が、なければ無しで構わない。

```
────────────────
【スマートフォン専用レイアウトの指示（最優先・厳守）】
このセクションは【スマートフォン縦画面専用】のセクション画像として出力する。
・左右2カラムの要素は、必ず縦1カラムに積む（左右に並べない）
・横並びのアイコン・カード・ボタンが3つ以上ある場合は、2列グリッドまたは縦積みに構成する（横一列にしない）
・見出し・本文とも、スマートフォンで実寸表示した際に十分大きく読める文字サイズにする
・縦横比は9:16前後の縦長を基本にする（750px幅程度のスマートフォン縦型LPセクションとして出力する）
・上下端は前後セクションへ自然につながる余白で終える
```

出力先: `lp/images/mobile/<section.imageName>`

このスマホ版が、上記「PC版をスマホ版基準で作り直す」の参照元になる。

**生成した画像は必ずReadツールで開き、下端に次セクションの要素が見切れて写り込んでいないか確認する。** 見つかったら下端をクロップして直す（AI画像生成はセクション境界を厳密に切れないことがある）。

## Step 3 — 結合しない（画像は1セクション1ファイルのまま使う）

旧方式にあった「全セクションを縦結合して`lp-full.png`/`lp-full-mobile.png`を作る」ステップは行わない。`lp/images/<section.imageName>`（PC）と `lp/images/mobile/<section.imageName>`（スマホ）を、そのまま個別の画像ファイルとしてHTMLから参照する。

## Step 4 — HTMLをセクションごとのpicture要素で組み立てる

`lp/index.html` を、`sections` の並び順どおりに1セクション＝1つの`<picture>`として書き出す。参考実装は `portfolio/astella-vxr-suv-lp/lp/index.html`。

```html
<main class="lp-shell" aria-label="<project.title>">
  <picture class="lp-section lp-section-hero">
    <source media="(max-width: 767px)" srcset="./images/mobile/01-hero.png">
    <img class="lp-section-img img-ratio-hero" src="./images/01-hero.png" alt="<project.title>" decoding="async" fetchpriority="high">
  </picture>
  <picture class="lp-section reveal">
    <source media="(max-width: 767px)" srcset="./images/mobile/02-xxx.png">
    <img class="lp-section-img img-ratio-xxx" src="./images/02-xxx.png" alt="" loading="lazy">
  </picture>
  <!-- 以下、セクション数だけ繰り返す -->
</main>
```

- 先頭セクション（ファーストビュー）だけ `fetchpriority="high"` を付け `loading="lazy"` は付けない。alt はページタイトル。それ以外のセクションは `loading="lazy"`、alt は空文字でよい（装飾的な画像のため）。
- 先頭セクションには `lp-section-hero` クラス（フェードイン用）、それ以外には `reveal`（スクロールで下から表示、Intersection Observerで `is-visible` を付与するJSが必要 — `astella-vxr-suv-lp` 参照）を付ける。
- `<img>` に `img-ratio-<section.id>` のようなセクション固有クラスを付け、`style.css` 側で `aspect-ratio` を **PC用（デフォルト）とスマホ用（`@media (max-width: 767px)`）の両方、生成した画像の実測縦横比で** 指定する（画像読み込み前のレイアウトシフトを防ぐ）。`sips -g pixelWidth -g pixelHeight <path>` 等で実測してから書く。
- `.lp-shell` は `max-width: 1200px; margin: 0 auto; width: 100%;`。`.lp-section-img` は `display: block; width: 100%; height: auto;`。

出来上がったら `lp/style.css` を `portfolio/astella-vxr-suv-lp/lp/style.css` と見比べ、`.lp-shell` / `.lp-section` / `.lp-section-img` / `.reveal` / `aspect-ratio` 一式が揃っているか確認する。

`reveal`（スクロールで下から表示）と「トップへ戻る」ボタンの動作には `lp/script.js` が必要（IntersectionObserverでの `is-visible` 付与、スクロール量での表示切替）。`portfolio/astella-vxr-suv-lp/lp/script.js` をそのままコピーして使ってよい。`index.html` の `</main>` 直後に `<button type="button" class="back-to-top" aria-label="トップへ戻る">↑</button>` を置き、`</body>` 直前で `<script src="./script.js"></script>` を読み込む。

## Step 5 — 動作確認

- ローカルで `python3 -m http.server` 等を立てて開き、ブラウザ幅を768px前後で切り替えて画像が変わることを確認する（自動化ツールのウィンドウリサイズが効かない場合は、ユーザーに実ブラウザでの確認を依頼する）。
- Vercelにデプロイ済みの場合は、Preview URLで確認してから本番反映する。

## Step 6 — ギャラリーでのプレビュー確認

`gallery/` サイトのカードをクリックすると、PC/スマホ切り替えボタン付きのプレビューモーダルが開く（`gallery/script.js` の `openPreview` / `setDevice` 実装済み）。新しいLPをギャラリーに追加した際は、このプレビューでも表示が崩れていないか確認する。
