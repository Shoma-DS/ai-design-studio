# WP GSAP Animate

WordPressの好きな要素にGSAPアニメーションを付ける。

**このスキルの核心は「ユーザーはCSSセレクタを知らなくていい」こと。**
動きは見本帳から名前で選んでもらい、セレクタはAIが実サイトのHTMLを読んで特定する。

## 頼まれ方（この3つが揃えば実装できる）

ユーザーはこう書いてくる。

```
/wp-gsap-animate section-color /test2  毎日のニュースを受け取るにはサインアップしてください、のセクション
/wp-gsap-animate split-lines /about    「私たちについて」の見出し、少しゆっくりめで
```

| 要素 | 例 | 必須か |
|---|---|---|
| **動きの名前** | `section-color` | 必須（見本帳の名前） |
| **ページ** | `/test2` | **必須**（どのページのHTMLを読むか決まらない） |
| **対象の説明** | 「〜のセクション」 | 必須（**中身のテキストで指してよい**） |
| **速さ・強さ** | 「少しゆっくりめで」 | **省略可**。対象の説明のあとに足される |

速さの指定は**文末に付いてくる**ことが多い。対象の説明と混ざらないよう、
「ゆっくり」「サッと」「大げさに」「控えめに」といった語は
**動きの調整（options）**として解釈し、要素を探す手がかりには使わないこと。

**対象はテキストで指してもらう。** これがこのスキルの前提。
ユーザーはCSSセレクタを知らない。「毎日のニュースを〜のセクション」と言われたら、
そのページのHTMLからそのテキストを含む要素を探し当てるのがAIの仕事。

**足りないものがあれば、足りないものだけ聞く。**

- 動きが無い → 見本帳を出して選んでもらう
- ページが無い → 「どのページですか？」とだけ聞く（トップなら `/`）
- 対象が無い → 「どの部分ですか？」とだけ聞く

全部揃っていれば、**何も聞かずに実装する**。

## 進め方の基本方針：まず動かす。確認で止まらない。

このスキルの変更は**取り消しがスニペット1つの無効化で済む**。投稿もテーマも一切変更しない。
つまり**間違えたときのコストがほぼゼロ**。だから、いちいち許可を求めて止まらない。

- **対象が明確なら、確認せずに実装して、結果だけ報告する。**
  「3カードに stagger を入れました。違っていたら『GSAP stagger サービスカード』の
  スニペットを無効化してください」
- **聞くのは、本当に絞れないときだけ。** 候補が複数あって区別できない、
  ユーザーの言う要素がページ内に見当たらない、など。
- **ただし次の2つだけは必ず守る。**
  1. 初めてそのサイトに入れるとき（＝FluentSnippetsのインストールが必要なとき）は一言断る
  2. 登録したあとは**必ずサイトを取得して 200 と error_files を確認**する

## 速度について（2回目以降は数秒で終わるはず）

**Coreスニペットは1サイトにつき1回だけ**。8.6KBあるので送るのに時間がかかる。
2回目以降のアニメーション追加では、**Coreの存在を確認して飛ばす**こと。
アニメ側のスニペットは1〜2KBしかないので、数秒で終わる。

**大きなコードをbase64にしない。** 生のコードのほうがトークン効率が良い。

## 前提の確認

- **対象サイトがNovamira(MCP)で接続されているか。**
  接続されていれば、HTMLの読み取りからスニペット登録まで全部自動でできる。
  接続されていなければ「手動モード」（後述）になる。

## 手順

### 1. 見本帳を見せる

ユーザーが動きの名前を言っていない場合は、まず見本帳を作って渡す。

スクリプトは**このスキルのディレクトリ**から実行する（`SKILL.md` がある場所）。
以降のコマンドはすべてそこが起点。

```bash
python3 scripts/build_gallery.py -o gallery.html
```

`gallery.html` を present_files で渡し、「気に入った動きの名前を教えてください」と伝える。
31種すべてが実際に動く状態で見られる。

すでに名前が分かっている場合（「split-lines で」）は、この手順を飛ばす。

### 2. 対象要素のセレクタを特定する（絶対に推測しない）

**必ず実サイトのHTMLを読む。** クラス名を推測すると動かない。

Novamiraが接続されている場合、`novamira/execute-php` でHTMLを取得する:

ユーザーが言ったテキスト（例:「毎日のニュースを受け取るには…」）を手がかりに、
**そのテキストを含む要素の周辺HTMLを切り出して構造を見る**のが確実。

```php
$url  = home_url( '/test2/' );   // ← ユーザーが指定したページ
$html = wp_remote_retrieve_body( wp_remote_get( $url, array('timeout'=>25) ) );

// ユーザーが言ったテキストの位置を探し、その前後を切り出して構造を読む
$needle = '毎日のニュースを受け取る';
$pos = strpos( $html, $needle );
if ( $pos === false ) {
    return array( 'not_found' => true );   // 言い回しが違う。別の語で再検索する
}
return substr( $html, max(0, $pos - 1500), 3000 );
```

切り出したHTMLから、囲んでいるセクション（`wp-block-group` など）と
その識別クラスを読み取る。「セクション全体」なら囲んでいるグループ、
「見出しだけ」ならその中の `h2` を選ぶ。

ページ全体の見出し一覧が欲しいときは:

```php
preg_match_all('/<h([1-3])[^>]*class="([^"]*)"[^>]*>(.*?)<\/h\1>/is', $html, $m, PREG_SET_ORDER);
```

**要素の中身のテキストで照合する。** ユーザーが言った「私たちのサービスの下の3つのカード」が、
HTML上のどの要素なのかを、テキストを突き合わせて特定する。

**一意に決まったら、確認を取らずにそのまま実装に進む。** 止まらない。
実装後の報告で「この要素に入れました（テキスト:「〇〇」）」と伝えれば足りる。

**聞くのは、絞れないときだけ。** 同じ見た目の候補が複数あって区別できない場合など。

#### セレクタの種類を見分ける

| クラス | 例 | 扱い |
|---|---|---|
| **意味のあるクラス** | `.wp-block-heading` `.wp-block-post-title` `.wp-block-column` | そのまま使う。安定している |
| **自動生成ハッシュ** | `.wp-container-core-columns-is-layout-794e3cfa` | **使ってよい。** ただし報告時に一言添える |

自動生成ハッシュは、**そのブロックのレイアウト設定を変更したときにだけ**変わる。
普段の記事編集では変わらないので、まずはこれで実装してよい。止まらない。

報告時にこう添える:

> 「なお、このカラムはWordPressが自動生成したクラスで指定しています。
> ブロックのレイアウト設定を変えると外れる可能性があります。
> 確実にしたい場合は、そのブロックの『高度な設定 → 追加CSSクラス』に
> `services-cards` と入れて教えてください。差し替えます。」

ユーザーがクラスを付けてくれたら、そちらに差し替える。

### 3. 構造要件をチェックする

`animations.json` の各アニメーションには `requirement` がある。

| requirement | 意味 | 実装前に確認すること |
|---|---|---|
| `any` | どこにでも付く | なし |
| `text` | テキスト要素向け | 文字が入っているか |
| `multi` | **子要素が2つ以上必要** | 直下に複数の子があるか。無ければ `children` で明示する |

条件を満たさないなら、**実装せずに先に伝える**。
（例:「このセクションは子要素が1つなので stagger は効きません」）

**レイアウトを壊すアニメーションは意図的に入れていない。**
画面固定や横スクロール（pin / horizontal-scroll / stacked-cards）は、
テーマCSSと干渉して崩れる・スマホで壊れるため、このスキルからは外してある。
「全部の動きが、要素を1つ指定すれば必ず付く」を保証するため。
ユーザーに求められても、**無理に自作しない**。崩れる原因になる。

### 4. 「どのページで動かすか」を決める（忘れやすい・重要）

ユーザーが「**トップページの**見出し」「**/test/ の**カード」と言ったら、それは
**そのページだけ**という意味。何も指定しないとサイト全体に効いてしまい、
他のページの同じクラスの要素まで動く。

ユーザーが特定のページを指していたら、**聞かずに `scope` を設定する**。
どことも言っていない場合だけ「全ページですか？」と確認する。

`scope` で限定する。

| scope | 適用先 |
|---|---|
| `all`（既定） | サイト全体 |
| `front` | トップページのみ |
| `home` | 投稿一覧ページのみ |
| `single` | 個別投稿のみ |
| `page` | 固定ページのみ |
| `page:12` | 固定ページID 12 のみ |
| `post:34` | 投稿ID 34 のみ |

### 5. 速さ・強さを言葉から読み取る

ユーザーは数値では言わない。聞き返さず、下記を目安に `options` に変換して、まず作って見せる。

| 言われ方 | 変換 |
|---|---|
| ゆっくり / じっくり | `duration: 1.4` |
| 少しゆっくり | `duration: 1.2` |
| サッと / キビキビ / スッと | `duration: 0.45` |
| 大げさに / 派手に | `distance: 80` |
| 控えめに / さりげなく | `distance: 20`, `duration: 0.6` |
| 一つずつゆっくり | `stagger: 0.2` |
| 少し遅れて / ワンテンポ置いて | `delay: 0.3` |
| ちょっと待ってから | `delay: 0.6` |

`delay` は**全アニメーション共通**で使える（動き出すまでの待ち時間・秒）。既定は 0。
「スクロールしてすぐ動くと唐突」というときに 0.2〜0.4 くらい入れると落ち着く。

**表現が矛盾していたら**（例:「スッと迫り上がる、少しゆっくりめ」）、
速さの直接指定（「ゆっくりめ」）を優先する。「スッと」は動きの質の形容であって
速さの指定とは限らない。迷ったら既定値のまま作り、見せてから調整する。

### 6. 設定ファイルを書く

`config.json` を作る。`name` は日本語でよい（スニペットの見出しになる）。

```json
{
  "animations": [
    { "name": "トップの見出し", "scope": "front",
      "selector": "main h1.wp-block-heading",
      "type": "text-mask", "options": { "duration": 1.1 } },
    { "name": "記事一覧", "scope": "front",
      "selector": ".wp-block-post-template",
      "type": "stagger", "children": ".wp-block-post-template > li" }
  ]
}
```

### 7. スニペットを生成する

```bash
python3 scripts/build_snippet.py --config config.json --out-dir snippets/
```

出力先の古いスニペットは自動で消える（残骸を登録する事故を防ぐため）。

出力される:

- `snippets/00-gsap-core.php` … GSAP読み込み＋共通の土台。**1つだけ登録する**
- `snippets/01-*.php` … アニメーション1つにつき1ファイル
- `snippets/README.txt` … 登録手順

**要素ごとに独立している**ので、「この見出しの動きだけ消したい」＝そのスニペットを無効化するだけ。

生成できたら、**そのまま次（登録）へ進む。ここで止まらない。**

### 8. WordPressに入れる

#### Novamiraが接続されている場合

1. **FluentSnippetsを入れる**（無料・ファイルベース・PHP/JS/CSS対応）:

   ```
   novamira/run-wp-cli  →  plugin install easy-code-manager --activate
   ```
   （スラッグは `easy-code-manager`。`fluent-snippets` ではない）

2. **Coreの「バージョン」を確認する。存在確認だけでは不十分。**

   Coreは1サイトにつき1つ。毎回送り直すと8.6KBぶんのトークンを書き出すことになり、
   体感で2分近く無駄になる。**が、「あるからスキップ」だけでは古いCoreが残り続け、
   修正済みのバグが再発する**（実際に、日本語の文字分割バグを直したのに
   古いCoreのままで text-fill がただのフェードインになる事故が起きた）。

   `snippets/00-gsap-core.php` の先頭にある `wpgsap-core-version:` と、
   サイト側のCoreスニペットの同じ行を**突き合わせる**。

   ```php
   $dir = \FluentSnippets\App\Helpers\Helper::getStorageDir();
   $found = array();
   foreach ( glob( $dir . '/*gsap-core*.php' ) as $f ) {
       $c = file_get_contents( $f );
       preg_match( '/wpgsap-core-version:\s*([\d.]+)/', $c, $m );
       $found[ basename( $f ) ] = isset( $m[1] ) ? $m[1] : '0.0.0';  // 版が無ければ最古扱い
   }
   return $found;
   ```

   | 状況 | やること |
   |---|---|
   | Coreが無い | 新規に1つ登録する |
   | 最新版が1つだけある | **何もしない**（送り直さない。これが速い） |
   | 古い版がある | **古いファイルを削除して、新しいCoreを登録し直す** |
   | 複数ある（重複） | **1つだけ残して、他は削除する** |

   Coreを入れ替えたら `Helper::cacheSnippetIndex( '', true )` を忘れずに。

   **アニメーション側のスニペットにも同じ問題がある。**
   プリセットのコードは各スニペットに**焼き込まれている**ので、
   スキル側でプリセットを直しても、既にサイトに入っているスニペットは古いまま。
   `wpgsap-preset-version:` を見て、古ければ作り直す。

   ```php
   foreach ( glob( $dir . '/*gsap-*.php' ) as $f ) {
       $c = file_get_contents( $f );
       preg_match( '/wpgsap-preset-version:\s*([\d.]+)/', $c, $m );
       // 版が無い or 古い → そのスニペットを作り直す
   }
   ```

   ユーザーから「直したはずの動きが直っていない」と言われたら、
   **まずこの2つ（Coreとスニペットの版）を疑うこと。**

3. **プラグイン自身のAPIで登録する。** ファイル形式を手で組み立てない。

   **base64にしない。** エスケープを避けたくなるが、base64はランダム文字列で
   トークン効率が最悪（生コードの倍かかる）。PHPコードは**ヒアドキュメントで
   そのまま渡す**こと。終端記号は本文と衝突しないもの（例 `WPGSAP_SNIPPET`）を使う。

   ```php
   $code = <<<'WPGSAP_SNIPPET'
   <?php
   ...ここに build_snippet.py が出力したPHPをそのまま貼る...
   WPGSAP_SNIPPET;
   ```

   ```php
   use FluentSnippets\App\Model\Snippet;
   use FluentSnippets\App\Helpers\Helper;

   $model = new Snippet();
   $fileName = $model->createSnippet( $code, array(
       'name'     => 'GSAP Core',
       'status'   => 'published',   // 'draft' だと動かない
       'type'     => 'PHP',
       'run_at'   => 'all',         // PHPスニペットは 'all' か 'backend' のみ
       'priority' => 5,             // Coreを5、各アニメを10にする
       'group'    => 'GSAP',
   ) );
   Helper::cacheSnippetIndex( '', true );   // インデックス再構築。忘れると反映されない
   ```

   **必ず `00-gsap-core.php` を先に（Core未導入のときだけ）登録する。**

   **【最重要】`$code` は `<?php` を含めたまま渡すこと。**
   FluentSnippetsが前置きするメタ情報は `?>` で終わる。つまり本文は
   HTMLモードから始まる。`<?php` を外して渡すと**コードがHTMLとして出力され、
   途中の `<?php` から構文が壊れてサイトが500になる**。
   `build_snippet.py` の出力をそのまま（先頭の `<?php` ごと）渡せばよい。

4. **登録したら、必ず動作確認する。飛ばさない。**

   ```php
   $res = wp_remote_get( home_url('/'), array('timeout'=>30) );
   $code = wp_remote_retrieve_response_code( $res );      // 200 か
   $cfg  = Helper::getIndexedConfig( false );
   $errs = $cfg['error_files'];                           // 空か
   ```

   - **500になっていたら、即座にスニペットを削除して復旧させる**
     （`wp-content/fluent-snippet-storage/` の該当ファイルを消す）
   - `error_files` に入っていたら構文エラー。中身を直す
   - 保存先は `wp-content/fluent-snippet-storage/`

   なおFluentSnippetsは致命的エラーを検知したスニペットを自動で無効化するので、
   サイトが真っ白のままにはなりにくい。それでも確認は必ずすること。

5. ページのHTMLを取得して、実際にコードが出力されているか確かめる:
   `root.WPGSAP` と `gsap.min.js` が入っていれば読み込まれている。
   （動作確認とHTML確認は**1回のexecute-phpにまとめる**。往復を増やさない）

6. ユーザーにページを再読み込みしてもらい、動きを確認する。

#### Novamiraが無い場合（手動モード）

生成したスニペットのコードをそのままユーザーに渡し、こう案内する:

1. FluentSnippets（またはWPCode）をインストール
2. New Snippet → **Functions (PHP)** を選ぶ
3. `00-gsap-core.php` の中身を貼って、Run on = Everywhere で有効化
4. 残りのスニペットも同様に登録

セレクタが分からない場合は、ブロックエディタの
「高度な設定 → 追加CSSクラス」に任意のクラス（例 `anim-hero`）を
入れてもらい、それをセレクタにする。

### 9. 確認と微調整

- 動かないときは**ブラウザのコンソール**を見てもらう。
  `[wp-gsap] 要素が見つかりません: .xxx` と出ていればセレクタが違う
- 「もっとゆっくり」等は `config.json` の `options` を変えて再生成するだけ

## 守ること

- **セレクタを推測しない。** 必ず実際のHTMLを読む（読んだ結果で判断するのは可）
- **登録したら必ずサイトを取得して確認する。** 500やerror_filesを放置しない
- **`prefers-reduced-motion` を尊重する。** 対応済み。無効化しない
- **やりすぎない。** 1ページ3〜5箇所までが上品。全部に付けると安っぽくなる

## 報告のしかた（実装後）

止まらずに実装したぶん、報告は具体的にする。

- どの要素に、何を入れたか（テキストで照合できる形で）
- どのページに効くか
- **取り消し方**（そのスニペットを無効化するだけ、と明示する）
- ハッシュクラスを使った場合はその注記

## リッチテキストの注意

対象要素の中に `<strong>` `<a>` などが入っていることがある。

- `split-chars` / `split-words` / `typewriter` … タグを保ったまま分割する（安全）
- `split-lines` … 行マスクを作るために組み替えるので、**中のタグは失われる**。
  装飾を残したい場合は `split-words` か `text-mask` を使う（コンソールに警告が出る）

## 元に戻す方法

- 特定の動きだけ止める → そのスニペットを無効化
- 全部止める → `00-gsap-core` を無効化
- サイトの投稿内容やテーマには一切手を加えていないので、痕跡は残らない

## 仕組み

- GSAP + ScrollTrigger をCDNから読み込む。**GSAPは2025年に全機能無料化**（商用利用可）
- `engine/core.js` … 共通の土台（オプション解決・ScrollTrigger・テキスト分割・FOUC解除）
- `engine/presets.js` … 31種の実装。`/* @preset <id> */` 〜 `/* @endpreset */` で囲まれている。
  `build_snippet.py` はこのマーカーで**使う分だけ切り出す**。マーカーを壊さないこと
- **見本帳と本番は同じコードで動く**（どちらも core + presets から生成される）
- テキスト分割はSplitText非依存の自前実装
- 他のプラグインが既にGSAPを読んでいれば二重読み込みしない
- FOUC対策済み。各スニペットが `<style class="wpgsap-fouc">` を1枚出し、
  Coreがまとめて外す。JSが落ちても3秒後に必ず表示が戻る
